# 更新重复日程

最后更新：2022/09/28

---

# 概述

企业微信重复日程（即周期性日程）支持多种修改模式。

用户可能只希望：

* 修改某一次日程
* 修改未来所有日程
* 修改全部日程

因此 API 提供了三种更新模式。

> 删除日程与更新日程逻辑类似，下文以“更新日程”为例说明。

调用者需配合以下两个参数：

* `op_mode`
* `op_start_time`

实现不同的修改方式。

---

# 三种修改模式

## 1. 全部修改（默认模式）

修改所有周期的日程。

即：

```text id="4e3k7y"
整个重复日程序列全部更新
```

这是默认行为。

---

## 2. 仅修改此日程

仅修改指定周期的某一次日程。

系统行为：

1. 原重复日程删除指定周期
2. 自动生成一个新的单次日程
3. 更新接口中的参数作用于新日程

即：

```text id="9k2vtb"
原周期性日程 A
→ 删除其中一次周期
→ 分裂生成单次日程 B
```

---

## 3. 修改将来的所有日程

修改指定周期及之后的所有周期。

系统行为：

1. 原日程的 `repeat_until` 被修改
2. 原日程被修剪
3. 从指定周期开始分裂出新的重复日程

即：

```text id="7v9hde"
原日程 A 被截断
→ 生成新的重复日程 B
→ 新参数作用于 B
```

---

# 修改原理

不同模式下：

| 模式        | 是否分裂新日程 | 是否修剪原日程 |
| --------- | ------- | ------- |
| 全部修改      | 否       | 否       |
| 仅修改此日程    | 是       | 是       |
| 修改将来的所有日程 | 是       | 是       |

---

# 注意事项

## 1. 只有重复日程才支持特殊修改模式

以下模式：

* `仅修改此日程`
* `修改将来的所有日程`

仅适用于：

```text id="9t4n1f"
周期性日程
```

如果普通单次日程指定这些模式：

```text id="g8o7wu"
会报错 90485
```

---

## 2. 必须指定 op_start_time

当：

```text id="v1aqeo"
op_mode = 1 或 2
```

时：

必须指定：

```text id="clqxy6"
op_start_time
```

用于指定：

```text id="3m7t0y"
从哪个周期开始修改
```

---

## 3. op_start_time 必须精确匹配某个周期

要求：

```text id="yax2zg"
op_start_time 必须是原日程某个周期的开始时间
```

格式：

```text id="v1i2b5"
Unix 时间戳（精确到秒）
```

如果后台无法匹配：

```text id="twz9v7"
报错 90482
```

---

## 4. 企业微信会自动分裂新日程

当：

```text id="n4dzy0"
op_mode = 1 或 2
```

时：

企业微信会：

1. 修剪原日程 A
2. 分裂生成新日程 B
3. 更新参数作用于 B

新日程 ID：

通过返回参数：

```text id="2j9p6s"
schedule_id
```

返回。

---

## 5. 特殊情况：修改未来全部 + 首周期

如果：

```text id="x0g7jm"
op_mode = 2
```

并且：

```text id="zruv49"
op_start_time == start_time
```

即：

```text id="d8s3k1"
从第一个周期开始修改
```

则系统会自动退化为：

```text id="h6e1rt"
修改全部日程
```

此时：

* 不会分裂新日程
* 不会生成新的 `schedule_id`

---

## 6. “仅修改此日程”不能继续重复

当：

```text id="9r0m7o"
op_mode = 1
```

时：

新生成的日程 B：

```text id="j4zv3e"
只能是单次日程
```

因此：

```text id="q5bp4x"
is_repeat 不能为 1
```

否则：

```text id="jz1kl4"
报错 90484
```

同时：

* `repeat_type`
* `repeat_interval`
* `repeat_day_of_week`
* `repeat_day_of_month`

等重复参数也会失效。

---

# 案例分析

## 学校课程表场景

学校教师通过企业微信日历 API：

1. 创建课程表日历
2. 添加体育课重复日程

---

## 初始课程设置

体育课：

```text id="j2t4x8"
每周三 14:00
```

第一次上课时间：

```text id="0k0p9f"
2022-09-07 14:00:00
```

对应：

```text id="r7v8m3"
start_time = 1662530400
```

重复类型：

```text id="2e0zdx"
每周重复
```

于是：

```text id="y6k3la"
企业微信会每周三提醒学生上课
```

---

## 第二周临时调课

第二周：

体育老师临时有事。

课程改为：

```text id="3m4oky"
16:00 上课
```

原时间：

```text id="dc5m7t"
2022-09-14 14:00:00
```

对应：

```text id="i7m3zn"
1663135200
```

新时间：

```text id="v7g4sk"
2022-09-14 16:00:00
```

对应：

```text id="a0xw1r"
1663142400
```

---

## 更新方式

调用更新接口时：

```json id="p7x1yv"
{
  "op_mode": 1,
  "op_start_time": 1663135200,
  "schedule": {
    "start_time": 1663142400
  }
}
```

含义：

| 参数              | 作用       |
| --------------- | -------- |
| `op_mode=1`     | 仅修改当前周期  |
| `op_start_time` | 指定第二周课程  |
| `start_time`    | 设置新的开始时间 |

最终效果：

```text id="6d1g8n"
只有第二周课程变为 16:00
其它周次保持不变
```
